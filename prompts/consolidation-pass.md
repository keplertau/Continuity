# Consolidation Pass Prompt

This is a recipe an AI assistant can follow to run a consolidation pass on a Continuity layer. It implements the **Sleep-Cycle Distillation** pattern from SPEC §14.1.

The recipe is portable across substrates: it can run at session end as a manual step, on a schedule (overnight, weekly), or through an AI memory process. The pattern is what matters; how it executes is an implementation choice.

A pass that has nothing to do should write nothing. The pass must be quiet when nothing has changed.

---

## When To Use This Prompt

A consolidation pass is most useful when a Continuity layer has accumulated material since the last pass: new transcripts, new sessions, new decisions, new evidence. For a Lite or Standard setup with light activity, a manual end-of-session update is usually enough; a full distillation is overkill. For a Project-mode workspace with regular session activity, running this pass periodically tends to outperform real-time curation.

If you are the assistant running this for the first time on a layer, do a careful, conservative pass and surface what you did so the human can correct before it becomes habit.

---

## The Prompt

```text
Run a consolidation pass on my Continuity layer.

Follow the Sleep-Cycle Distillation recipe from SPEC §14.1. The pass has five phases: extract, distill, graduate, promote, log. If a phase has nothing to do, write nothing for that phase.

Hard rules:
- Non-destructive. Never delete files. Never overwrite transcripts or other evidence records.
- Briefings may be overwritten because they are derived files.
- No interpretation. Preserve transcripts verbatim. Summarize decisions accurately. Do not resolve ambiguity by inferring what someone meant.
- No forced output. If no session, concept, or decision qualifies for auto-graduation, write nothing.
- Provisional decisions must not flow into the decisions briefing until promoted. This is the safety rail that lets the pass run without supervision: it cannot reshape operating stances by inference alone.
- Compression discipline. Briefings loaded at session start must carry operational signal, not archival completeness. Prefer source pointers over rationale.

Locate the Continuity layer first. Read the index file (or the equivalent orientation file) before doing anything. Read the existing briefings to understand current state.

Then proceed through the five phases below.

---

PHASE 1: EXTRACT

Goal: pull new session transcripts (or chat exports, or notes) into the evidence layer with explicit fidelity labels.

Steps:
1. Identify what has been added since the last pass — transcripts, chat exports, notes, source documents, anything that should be preserved as evidence.
2. Skip meta-sessions: any session that is itself a consolidation pass, a maintenance task, a journal regeneration, or otherwise about the system rather than the work.
3. Deduplicate against a state record of previously extracted material. The state record can be a JSON file, a list at the top of an extraction log, or any other inspectable form. The discipline is that the pass is idempotent: running it twice on the same input produces the same output.
4. For each new transcript or extract, write it to the evidence layer with a header that declares fidelity:

   - **Extraction fidelity: full verbatim** — the entire record is preserved exactly as it appeared.
   - **Extraction fidelity: partial** — part of the record is missing. Add a short note immediately under the header naming what is missing and why.
   - **Extraction fidelity: compressed** — sections are summarized rather than verbatim. Add a short note naming what is summarized and why.

5. Preserve role prefixes (`[user]`, `[assistant]`, `[system]`) when extracting transcripts. They carry attribution that summaries lose.

6. If an oversized transcript cannot be extracted fully because of tool, host, or size limits, do not silently present a reconstruction as verbatim. Use `partial` or `compressed` and name the constraint.

7. Update the state record after each successful extraction.

---

PHASE 2: DISTILL

Goal: regenerate derived briefings from the underlying entities and recently extracted records.

Briefings are derived, not authored. They may be overwritten. They must remain compact.

Steps:
1. Identify which briefings exist in the layer. A typical Project-mode setup has at least:
   - a current-state briefing (what's in motion, what landed, what's open)
   - a decisions briefing (active operating stances, plus on-hold and superseded sections)
   - a recent-sessions briefing (rolling 4–6 week summary)
   - optionally a process briefing (recurring cadences)

2. Manually authored briefings are not regenerated by the pass. The roadmap and the project structure briefing (if one exists) belong to the human. Do not overwrite them.

3. For each derived briefing:
   a. Read the underlying entities (decisions, sessions, concepts, evidence) and recently extracted transcripts.
   b. Synthesize the briefing in the structure already established by the existing file. Preserve section headings and layout.
   c. For the decisions briefing, include only entities with `status: active`, `status: confirmed`, or no status field. Skip `status: provisional` entities — they belong in the recent-sessions briefing's auto-graduated section, not in the operating-stance register.
   d. For the recent-sessions briefing, surface auto-graduated provisional entities at the top in a labeled block, so the human can confirm or revise framings.
   e. Compression rule: if an entry takes more than three sentences, shorten it unless the extra detail is required to prevent operational error.

4. Preserve any inline status markers from SPEC §9 that carry signal beyond what section placement provides. The most common useful inline marker in derived briefings is `[STALE - REVIEW]` on open threads dormant beyond a project-defined threshold. Do not blanket-apply other markers — section placement and the auto-graduated block already carry that signal.

5. Write the regenerated briefing to its file. Do not change the substance of decisions; you are summarizing and organizing, not interpreting. If a decision is ambiguous, preserve the ambiguity rather than resolving it.

---

PHASE 3: GRADUATE

Goal: write new entity pages directly for candidates surfaced in recent material.

Scope:
- session entities (one per meaningful session that has a transcript but no summary)
- concept entities (named ideas used multiple times across recent sessions)
- decision entities (operational stances established in transcript but not yet recorded)

Do not auto-graduate entities outside this scope — people, tools, fragments, works, and other entity categories require human authoring.

Steps:
1. For each session that has a new transcript but no corresponding session entity:
   - Write a session entity with metadata (date, agent, tags, related entities, transcript pointer) and content sections (what happened, files modified, decisions made, pending items).
   - Session entities carry no `status` flag — they are records, not rules.

2. For each concept candidate (a named idea used multiple times across recent transcripts with no existing entity):
   - Write a concept entity with `status: provisional`.
   - Note that the framing is to be confirmed on first substantive next use.

3. For each decision candidate (an operational stance established in a transcript but not yet recorded):
   - Write a decision entity with `status: provisional`.
   - Use conservative metadata. If a field is unclear, leave it blank rather than inferring.
   - Provisional decisions must not propagate into the decisions briefing until promoted.

4. If nothing qualifies, write nothing. Do not force candidates.

---

PHASE 4: PROMOTE

Goal: promote provisional entities to confirmed when later session activity supports them.

Steps:
1. For each existing entity with `status: provisional`, check newer transcripts (post-creation date) for evidence of use:
   - The entity's slug, title, or distinctive phrase appears in a later session.
   - The usage does not object to or revise the entity.

2. If both conditions hold, promote the entity by changing `status: provisional` to `status: confirmed`. Note the promotion in the consolidation log.

3. If the logic is ambiguous — the slug appeared but the discussion may have been revising the framing — leave the entity provisional. Surface it to the human in the next session rather than auto-promoting.

4. Provisional entities that never get used are a normal outcome. Periodic merge/prune passes (SPEC §11.8) may consolidate or retire them.

---

PHASE 5: LOG

Goal: append an audit-trail record of what the pass did.

Steps:
1. Append a record to a consolidation log. The log lives wherever the project records process-runs — commonly under the processes folder, or alongside the briefings, or as a dedicated `consolidation-log.md`.

2. The log entry should record:
   - Date and time of the run.
   - Transcripts extracted (count and filenames).
   - Briefings regenerated (which ones).
   - Entities auto-graduated (count by category).
   - Entities promoted from provisional to confirmed.
   - Any extraction-fidelity issues, partial transcripts, or anomalies.
   - Any errors or skipped items.

3. The log is the audit trail for an otherwise-silent process. Future humans and future AI sessions can read it to understand what the consolidation pass has been doing.

4. If the pass had nothing to do — no new transcripts, no graduation candidates, no promotions — write a short "no-op" log entry rather than nothing. The log itself should not lie about whether the pass ran.

---

When the pass is complete, surface a brief summary in the chat: which phases produced output, what was added, what was promoted, what was flagged for review. Do not perform completeness — if the pass was quiet, say so.
```

---

## Adapting The Prompt

This recipe assumes a markdown-folder substrate. Adapt as needed:

- **Other substrates** (Obsidian, Notion, wiki, managed agent memory): translate "files" and "folders" into the substrate's equivalents. Status flags may live in frontmatter, in tags, in database columns, or in some other form. The discipline (provisional gate, fidelity headers, derived vs authored, idempotent pass) is what carries.

- **Different cadence** (manual end-of-session vs scheduled overnight vs always-on agent): the recipe does not assume a specific schedule. A manual run at session end is the simplest form; an overnight scheduled task is a richer one. The phases are the same.

- **Tool requirements**: the recipe needs file-read and file-write capability against the Continuity layer. If a tool can also enumerate prior sessions or chat exports (for example, a `list_sessions` and `read_transcript` API), Phase 1 becomes more automated. If it cannot, Phase 1 may need the human to provide transcripts manually.

- **Lite or Standard setups**: the recipe is calibrated for Project mode. For lighter setups, simplify: a Lite layer may only need Phase 1 (capture what changed) and Phase 5 (log it) at session end. A Standard layer may add Phase 2 (regenerate the index file) and skip Phases 3–4 unless the project has an entity layer.

## When NOT To Run This

- The Continuity layer has not changed since the last pass.
- The human has explicitly asked for a focused update rather than a full pass.
- The pass would run on top of unfinished work the human has not yet committed to memory — wait for the human to close the loop first.

The pattern is: the consolidation pass is for distilling settled material. Live work belongs in chat and in `current.md`, not in the entity layer.
